# Plan: AgToosa v3.1.0 Release

**Branch:** main  
**Status:** APPROVED — /autoplan 2026-05-04  
**Date:** 2026-05-04  

---

## Problem Statement

The v3.1.0 foundations were shipped in commit `bf5f870` (ADR-001 through ADR-004 action items):
- File-type allowlist enforcement in `lib/registry.sh`
- `agtoosa-lock.json` writing on install/update
- `--registry publish` contribution wizard
- `agtoosa.ps1` brought to v3.0.0 parity
- CI version parity check (bash vs PowerShell)
- `template/Docs/AgToosa_Governance.md` phase-gate protocol
- Formal deprecation policy in `CONTRIBUTING.md`
- 9 new bats tests (122 total)

But the codebase has three closure gaps:
1. `AGTOOSA_VERSION` is still `3.0.0` in `agtoosa.sh`
2. `CHANGELOG.md` has no `[3.1.0]` entry
3. ADR action item checkboxes in `docs/adr/` still show `[ ]` for work that was completed

This plan closes those gaps and cuts the v3.1.0 release.

---

## Premises

- The work in `bf5f870` is correct and test-passing (122 bats tests passing)
- The version bump from 3.0.0 → 3.1.0 is appropriate (MINOR: new features, backward-compatible)
- Remaining open ADR items (interactive migration wizard, GitHub Action for manifest validation) are v3.1.x concerns, not blocking v3.1.0
- PowerShell version should be kept in sync with bash (CI now enforces this)

```
## [3.1.0] — 2026-05-04

### Added
- Repo-native maintainer guide in `docs/agtoosa-maintainer.md` for working on the AgToosa generator itself

### Changed
- `GEMINI.md` now contains maintainer guidance instead of remaining empty
- `.github/agents/agtoosa.agent.md` delegates to shared maintainer guide

- CI now asserts version parity between `agtoosa.sh` and `agtoosa.ps1`
- `--update` now bumps `agtoosa_version` in existing `agtoosa-lock.json`; first run on a 3.0.0 install creates `agtoosa-lock.json` automatically in the project root
- File-type allowlist rejection message: "Rejected: &lt;file&gt; — only .md/.json/.toml/.mdc are permitted"
```
